# policy-rag-assistant 深度拆解：一個「答錯時一眼抓得到」的法規 RAG

> **關於本文的寫法**：這篇刻意把兩個貢獻者分開標示、不混為一談——**Jack**（人：負責方向、架構紅線、成功定義、當裁判、驗收）與 **Claude**（AI：負責實作、測試、除錯、寫開發日誌）。把「AI 做了什麼、人做了什麼」說清楚，本身就是這個作品想傳達的事。
>
> 這是一個 **PoC**，沒有真實使用者、沒上線。語料全是公開政府法規＋去識別化/合成的公司辦法，還沒碰真實髒資料。但底下這套「治理層」是真的——每個設計、每個坑、每個取捨，下面都對得到具體程式碼。工具：Claude Code（Python 幾乎全由 Claude 實作）＋ Gemini（runtime 路由與生成）。本文也標明了哪些是程式碼可直接驗的硬證據、哪些只是開發日誌的文字宣稱。

---

## 一、這是什麼、為什麼做

一個繁體中文的「公司管理辦法＋政府法規」RAG 問答 PoC。使用者用口語問——例如「家裡突然有急事，明天沒辦法上班怎麼辦？」——系統會路由到相關辦法/法條、撈出原文、由 LLM 帶著出處作答，並在答案底部附上**由程式插入的條文原文、免責聲明、原始檔載點**。

它有三個目的疊在一起：Jack 學 RAG、當求職作品、以及給主管看的 demo。定位是「邊做邊學」的一站，不是要做上線產品。

**語料規模（實測，非估計）**：catalog 共 134 筆，其中 127 部政府法令（來源：全國法規資料庫、政府電子採購網），涵蓋勞動、職安衛、公司/商業登記、稅務、勞健保、智財、個資、環安、消防、食安等**多個領域**——不是單一主題；另有 5 個 doc_no、7 個版本檔的公司 HR／管理辦法（皆為去識別化或合成，用來示範版本鏈與時間盲區）。全語料合計 7,062 條。

## 二、成功定義：不是「答得出」，是「答錯時一眼抓得到」

這是 Jack 為整個案子訂的招牌主張，也是所有設計的源頭。

一般 RAG 追求「答得準」。但在**法規場景，「看起來對」比「查無」危險**——一個流暢、有條號、口氣篤定、實際上引錯條文的答案，會害人做錯決定，而且它偽裝得很好。所以 Jack 把成功定義從「答得出來」改成「**答錯的時候一眼就抓得到**」。

由這個定義，Jack 定下三條鐵則，貫穿全案：**一字不漏、不竄改法條、查無誠實**；以及一條安全哲學：**LLM 是答題機、不是守門員**——權限、範圍、長度、流量、原文正確性，全部由程式碼守，不交給模型自律。

## 三、分工：Jack 定方向，Claude 落地

**Jack（人）下的判斷/方向**（有 Jack 交付給建置 session 的起手 prompt 為據）：

- **架構紅線**：目錄路由 + 治理層，**不建向量 DB**。
- **版本守門機制**：用「生效日/失效日 + 參考日」動態過濾，不是二元的現行/廢止；新版公告不刪舊版；主動預告未生效的新版。
- **安全哲學**：LLM 是答題機不是守門員，合規由程式碼硬編碼。
- **輸出合規**：答條款附 verbatim 全文不省略，免責與載點由程式碼硬編碼、LLM 不可竄改。
- **工作方式與剎車**：一步一步走、每步做完給我看再走；不要過早優化；**eval 是強制驗收，測不過就是沒做完**；**該打臉就打臉，別順著我說好話——我要公正評價，不是情緒價值**。
- **戳出最大留白**：主動指出「生產端『裸文件→自動產 metadata』沒做，我們驗的是消費端」。

**Claude（AI）的實作**：主要程式全由 Claude 寫，並在多處派 sub-agent 分工（開發日誌有據）——7 個 sub-agent 平行抓 111 部法規、6 個 Sonnet sub-agent 平行生成摘要/tags、coder sub-agent 補抓 10 部勞基法系、以及在 Gemini 掛掉時派 sub-agent 當「生成替身」做測試。sub-agent 產出後，由主流程獨立驗條數、版本、來源分類、verbatim，Jack 再複審。

> 誠實註記：開發日誌是 Claude 代筆的自述，「0 失敗」「全過」這類話**沒有第三方存證**，屬中等可信。但程式碼本身（反查邏輯、schema 綁定、二段路由）是硬證據——所以本文多靠 code、少靠日誌口頭宣稱。

## 四、關鍵技術決策與取捨

每個決策都寫「問題 / 選項 / 決定（誰定） / 理由 / **代價**」——尤其代價，因為沒有代價的設計通常是還沒想清楚。

### 1. 不建向量，改結構化目錄路由（Jack 定）
- **問題**：RAG 的常規做法是切 chunk＋向量相似度檢索。這裡要不要建向量庫？
- **選項**：(A) 向量 DB＋embedding；(B) 結構化目錄＋LLM 路由。
- **決定**：Jack 定架構紅線——(B)，不建向量。
- **理由**：法規有**天然的條級單元**（第 X 條、章），本來就切好了，不需要再猜 chunk 邊界。數百部法規的「目錄視圖」（只有名稱/摘要/tags、不含內文）塞得進 context，讓 LLM 直接做語意路由即可。與其一開始上重型架構，不如**把升級路線寫清楚**。
- **代價**：規模天花板。目錄總有塞不下 context 的一天。維護文件裡留了明確接縫：「規模大到目錄塞不下、需要向量架構時，於此新增一節」——刻意的 YAGNI，不是忽略。實測 122 部的路由視圖約 8.8K token，還在 context 裡。

這是本案最容易被質疑的決定（「沒向量算什麼 RAG」）。Jack 的立場是：RAG 的核心是「先檢索、再讓模型接著檢索到的內容生成」，向量只是最常見的檢索手段、不是定義。對這批結構乾淨的資料，結構化路由又簡單又夠用；賭錯了，路線圖接得上去。

### 2. 二段窄餵（法→章→條），破萬字大法省 token（Claude 實作）
- **問題**：公司法 534 條、勞基法 98 條，整部餵進 context 又爆量又稀釋注意力。
- **決定**：Claude 實作 `narrow_fetched` 遞迴窄餵——條數少於門檻（預設 30）就整部餵；否則先路由到「章」、再到「條」。小法或無章結構的法整部餵，而且**任何一層子路由失敗都安全退回整部餵**（寧可多給、不要漏）。
- **效果**：一題「公司法董事會決議」，餵給生成模型的內容從整部（實測 560 條）縮到單一條文（第 206 條），答案仍正確、引用乾淨。
- **代價**：多了幾次 route 的 API 呼叫，疊加延遲（見「限制」）。Claude 評估過並行化，但免費層的 RPM/並發上限要另做分流、收益不確定，依 Jack「不要過早優化」的指示 YAGNI 不做。

### 3. route 與 generate 拆用不同 Gemini 型號（Claude 實作）
- **問題**：Gemini 免費層配額是 per-model、per-day，各型號額度還不同。單一型號同時扛路由＋生成會互相擠壓、很快撞牆。
- **決定**：Claude 把路由用較省的 `flash-lite`、生成用 `flash`。一題＝路由×1＋生成×1，分屬兩型號 → **每日配額互不擠壓**。
- **代價**：免費層仍只夠 PoC 低頻開發。後來 Claude 補了生成端 fallback 鏈：撞過載/配額就自動換備援型號。

### 4. response_schema 綁結構——但清楚它綁不住幻覺（Claude 實作，延續 Jack 的安全哲學）
- **問題**：LLM 輸出 JSON 常帶 markdown 圍欄、缺欄位、格式漂移。
- **決定**：路由與生成都用 Gemini 的 `response_schema` ＋ `temperature=0`，在 decoder 層物理保證輸出是合法、欄位齊全的 JSON。
- **最誠實的一句**：Claude 在程式碼註解裡寫了「**schema 只保證『結構』、不保證 citations『內容完整』**」——這是延續 Jack「LLM 不當守門員」的原則，在實作時進一步認清：schema 能綁格式、綁不住幻覺，所以另外建了程式反查（下一條）。認清一個工具的邊界，比誇它萬能重要。

### 5. 程式反查 citation ＋ 可見失敗 ⚠️（本案核心；Jack 定原則、Claude 實作）
- **問題**：就算 schema 綁了 citations 欄位，模型還是可能自報錯條號，或在答案裡提到某條、實際上那條根本沒被檢索到（幻覺）。
- **決定**：延續 Jack「不信 LLM、由程式碼守門」的原則，Claude 實作了 `_audit_prose`——**不信模型自報的 citations**，改從答案文字裡正則挖出每一個「○○法第 X 條」，逐一對照「這次實際 fetch 到的原文」：
  - 有接地但沒顯示 → 程式補貼原文；
  - 沒檢索到 / 條號不在範圍 → **標 ⚠️ 警告**，並印「下列引用未取得原文佐證，請人工查核、勿全信」。
- **為什麼是核心**：它把「答錯」從無聲變成可見——這正是 Jack「答錯時一眼抓得到」成功定義的兌現。
- **代價/限制（誠實）**：純隱式引用（「依第 13 條」沒帶法名）、或類稱在檢索結果裡不唯一時，仍解不出來——這種情況照樣警告（偏安全側），寧可誤報、不要放過。

### 6. 版本守門＝參考日過濾（三態，不凍結）（Jack 定機制、Claude 實作）
- **問題**：法規有版本鏈。用「現行/廢止」二元旗標，跨時間查詢就會錯（問「7 月之後」該回新版）。
- **決定**：Jack 定機制——catalog 只存生效日/失效日原值，**三態（生效中/已公告未生效/已廢止）由查詢時的參考日動態算**；參考日預設今天、可被問題日期覆蓋。新版公告不刪舊版，把舊版失效日設成新版生效日前一天（宣導期共存）；主動預告未生效新版。Claude 實作。
- **配套鐵則**：摘要**不准寫具體數值**（金額、日數、門檻），因為數值寫進摘要，LLM 會直接讀摘要作答、**短路掉版本守門**。數值一律留原文，逼系統走「按參考日撈正確版本原文」。

### 7. verbatim 原文 / 免責 / 載點 由程式碼硬編碼（Jack 定、Claude 實作）
- **決定**：Jack 定輸出合規紅線——LLM 只負責「解讀＋給引用指針」、**不逐字抄條文**；條文原文由程式從已 fetch 的檔案插入，免責固定字串，載點動態填該答案引用的檔案，全部 LLM 不可竄改。Claude 實作；這段曾 silently 失效過、已修，並因此進了回歸測試。

### 8. 公司辦法 ↔ 政府法令 並陳（Claude 實作）
- **問題**：員工問請假，公司辦法和上位法令（勞基法、勞工請假規則）該一起給並講清楚關係。
- **決定**：Claude 讓路由挑到公司辦法時一併拉同主題政府法令（反之亦然）；答題時**並陳兩者**，指出「法令是最低標準、公司規定不得低於法令」，但個案合法性提醒洽人資/法務、不逕自裁決。靠每份文件標的 `source_type` 驅動。
- **代價**：多拉文件 → 延遲與成本略升。

## 五、踩過的坑與根因診斷

這節是「工程深度」最有料的地方。**分工說明**：這些對抗測試、找根因、改 code 是 Claude 在建置 session 裡做的；Jack 的角色是**要求對抗測試＋eval 強制驗收，並複審、拍板修法方向**。每個坑都是「發現 → 診斷根因 → 治本 → 加回歸測試」，不是打補丁。

### 條標題誤切——全語料 167 個「假條」
- **症狀**：條數對不上（某法解析出 102 條、實際 98 條）。
- **根因（Claude 診斷）**：內文裡的**交叉引用**（行首「第十七條規定於本條…準用之」）被誤判成新條標題，還吃掉前一條結尾。
- **治本（Claude）**：條號正則加「條號後必須接空白或行尾」。重建後 45 部法規、共修掉 167 個假條（勞基法 102→98、公司法 568→534）；驗過被排除的全是交叉引用、無真條遺失。

### 責任制第 84-1 條「假查無」（附則漏撈）
- **症狀**：在 Jack 要求的對抗測試下，Claude 用「錯誤前提誘導」黑箱題（「責任制第 84-1 條…對吧？」）測，測出系統把一個真實存在的條文誤判成「查無」。
- **根因（Claude 診斷）**：二段窄餵的第二段靠「猜主題」選章，猜不到那條藏在**『附則』章**、又被問題直接點名的條號。
- **治本（Claude 實作、Jack 複審拍板方向）**：讓窄餵**強制納入問題點名的條號**（繞過主題猜測），並補強「第 X 條之 N」格式辨識。重測後正確打臉錯誤前提、引出原文。
- **關鍵**：修之前它雖答錯，但程式反查有標 ⚠️——這個坑反過來印證了「可見失敗」設計的價值。

### 程式反查「狼來了」誤報
- **症狀**：模型答案用自然簡稱/指代（「同法」「本辦法」「施行細則」）時，反查系統性誤報「未檢索到原文」——其實條文都有接地。**警報疲勞**：狼來了喊多了，真警報也沒人信。
- **根因（Claude 診斷）**：反查用全名比對，配不到簡稱和指代。
- **治本（Claude）**：加簡稱/指代解析器——去掉連接詞後，先試精確名稱/別名、再試唯一類稱、最後純指代（「同法」）回指前文最近解析過的法。誤報 5→1。
- **重要的克制**：解析器**只解「是哪部法」，「那條到底在不在檢索結果裡」仍由條號存在檢查把關**——絕不讓簡稱解析蓋掉真正的檢索漏失。品質檢查器自己也不能變成幻覺的幫兇。

### 連接詞「且」漏洞——eval 當場抓到，Claude 也更正了自己的日誌
- **症狀**：把反查回歸測試固化成自動化 eval 時，**eval 當場抓到**（Jack 堅持的「eval 強制驗收」在此發揮作用）：連接詞清單漏了「且」，導致「且依同法第 X 條」的指代解析失敗。
- **治本（Claude）**：補齊連接詞、把比對從行首錨定改成結尾錨定 → 6/6 通過。
- **加分細節**：Claude 在日誌裡**主動更正了自己前一筆的過度樂觀**——上一筆寫「某題已修」，這次發現當時其實沒修到、本次才真的修掉。開發過程會回頭糾正自己的紀錄（這是 Jack「eval 強制、不報喜」要求下的產物），比一路報喜重要。

### 模型作弊（用先驗知識）→ 反作弊 harness
- **症狀**：用 sub-agent 當「生成替身」測試時，替身可能用**自己的先驗法律知識**答題，而非只用檢索到的條文——這會讓測試「看起來過了」、實際沒測到檢索品質。
- **治本（Claude）**：替身**禁用先驗知識**（派工 prompt 強制），並用 grep 驗「答案裡每個數字都真的出現在餵進去的 prompt 裡」；冒出 prompt 沒有的數字，就是它在掰。檢索漏掉的，替身也看不到。

### Gemini 503 / 配額牆 → 探針 + 替身
- **症狀**：生成型號撞每日上限、備援型號又頻繁 503，端到端測試被卡住。
- **對策（Claude）**：寫一支**探針**對候選型號各打一次最小請求，分辨「額度為 0 / 已耗盡 / 可用」，把不確定變成可決策事實；生成端加 fallback 自動換模型；以及上面那套**不靠 Gemini 也能測**的替身 harness。

## 六、把 eval 當工程

**原則（Jack 定）**：夠不夠好一律用 eval 量、不用直覺；測不過就是沒做完。**設計與實作（Claude）**：

**題型**照「系統會怎麼壞」設計，每題驗三件事：①路由對不對 ②答案＋引用對不對 ③反唬爛（沒寫的別掰）。涵蓋：查無誠實、版本陷阱（該回現行版、不能回廢止版）、零關鍵字語意路由（「家裡有急事」要路由到請假）、防跨辦法污染（不能把出差緊急補單條款錯套到請假）、交叉引用沿圖補撈、對話追問改寫、時間盲區（答生效中版、預告未生效版）。

**工程化**分兩層：一層只驗路由＋版本守門，一層跑完整答題並自動斷言「必含/禁含」。

**球員兼裁判的規避**（本案亮點）：回歸測試不靠 LLM 判分，改用**確定性 fixture ＋ 程式斷言**；生成替身禁用先驗、grep 反作弊。讓「測試」和「被測系統」脫鉤——被測的東西不能同時當裁判。

> 誠實限制：「路由 8/8、答題 8/8」是開發日誌的**文字宣稱**，因為完整 eval 是即時打 API 的腳本、沒有留存 run 報告。**目前唯一能離線復現的是程式反查回歸測試（6 案、自帶 fixture、免 Gemini、任何環境可跑）**。作品集該靠這種可復現的東西，不是靠日誌上的漂亮數字。

## 七、誠實的限制與 tradeoff

放在明處，因為藏起來的限制遲早在面試或落地時反咬。

- **延遲**：約 13 秒/題（路由＋多次窄餵＋生成的 API 疊加）。而且這數字只有一句日誌、沒有 benchmark 存證——是概略值、不是實測基準。二段路由降的是 token 與成本，延遲其實是它換來的代價。
- **沒有真實使用者**：PoC，未上線。語料是公開法規＋去識別化/合成辦法，還沒碰真實髒資料。
- **沒用向量 DB**：規模天花板是已知的，升級接縫留好了。
- **合成辦法 vs 真辦法的結構風險**：eval 標準答案綁在合成辦法上，落地換真辦法要重新校準。分三層：內容層一定要換、能力層不隨資料變、**架構層——若真辦法結構（條文交互、版本鏈樣態）牽動引擎，就必須回頭改架構**。這是**已知風險，不是設計疏漏**。
- **最大留白：生產端攝取自動化沒做**（Jack 主動戳出）。metadata 目前手填 frontmatter；驗的是消費端（目錄→路由→答題），沒驗生產端（裸文件→自動分類/產 metadata）。
- **資料分級紅線**：免費層的輸入輸出可能被拿去訓練，所以**真實公司機密辦法絕不可走免費層**；PoC 一律只用公開/合成資料。後期已升付費層解除這條紅線，但真公司資料上雲仍是資料分級問題。

## 八、時間線

密集建置期約 3 天（第一天鎖 metadata schema 與版本守門、埋 eval 前置、建 catalog；第二天最密集——建四階段引擎、批量攝取 111 部法規、建二段路由與程式反查、修 167 假條、對抗測試修 84-1；第三天補齊勞基法系、修反查誤報、固化回歸測試），第四天收尾升付費層。

## 收尾：這個案子證明了什麼

它不證明 Jack 會做上線級的大 RAG——它是個 PoC，不誇大。它證明的是**判斷力與紀律**：Jack 定下「這種資料不必上向量」的紅線、「LLM 不當守門員、合規交給程式碼」的哲學、「eval 強制驗收、不報喜」的紀律、以及「把每個限制寫在明處」的誠實；Claude 把這些落地成能對到函式的程式碼，並在 Jack 要求的對抗測試與 eval 下，一個一個把坑診斷、治本、加回歸。

一句話：**Jack 賭的不是「AI 答得多準」，是「答錯的時候，系統證明得出它錯了」。** 在法規這種場景，後者才是能信任的前提。
